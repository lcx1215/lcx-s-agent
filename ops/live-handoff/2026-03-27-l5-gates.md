# L5 Gates

## Current judgment

- current phase: L4.3 -> L4.4 transition
- not yet L5

## What is correct

### 1. Clear the current live-port debt

This is required.

Open items:

- lane workspace propagation for learning council
- stronger dedupe for workface / scorecard / validation weekly

Reason:

- `dev-fixed but not live-fixed` debt blocks L5
- L5 requires repeated proof that live-port discipline is real, not just documented

### 2. Harden bookkeeping / memory failure handling

This is required.

Minimum acceptable target:

- `task result` and `bookkeeping result` are separated
- `memory_write_failed` is structured
- raw `Edit failed` style leakage is no longer the bookkeeping contract
- there is pending / retry / anomaly trace
- memory write failure does not silently erase continuity

### 3. Make at least two branches truly stable

This is required.

Priority order:

1. `technical_daily_branch`
2. `knowledge_maintenance_branch`
3. then consider `fundamental_research_branch`

Reason:

- `fundamental_research_branch` is heavier and easier to overgrow
- L5 should first prove that a lighter daily branch and a maintenance branch can run stably and repeatedly

### 4. Turn release / acceptance discipline into repeated behavior

This is required.

Minimum evidence:

- each patch has handoff
- each patch distinguishes `dev-fixed` and `live-fixed`
- each live patch has fixed Feishu acceptance phrases
- each live patch records files, build, restart, probe, and real Feishu verification
- this happens repeatedly, not just once

## What is not the path to L5

- adding more modules for appearance
- swapping models and calling that progress
- broad architecture expansion
- new provider/routing churn
- new memory architecture before bookkeeping is reliable

## L5 bar

Only claim L5 when most of these are true:

1. current live-port debt is mostly cleared
2. bookkeeping / memory failure is structured and no longer fragile file-edit semantics
3. at least two branches run stably, repeatedly, and with acceptance checks
4. release discipline is repeated consistently across multiple rounds
5. the system accumulates branch-level operating artifacts instead of only ad-hoc replies

## Immediate sequence

1. finish the two live-port tickets
2. harden bookkeeping / memory failure handling
3. stabilize two real branches
4. run several disciplined release / acceptance rounds
